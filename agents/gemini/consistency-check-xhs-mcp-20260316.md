## 二、矛盾清单

### 2.1 效率数字不一致

| 来源 | 效率提升数字 | 说明 |
|------|-------------|------|
| Gemini 扫描 | 50-70% | xhs-radar 效率提升预估 |
| NotebookLM 报告 | 60-80% | xhs-radar 效率提升预估 |
| **矛盾点** | 数字漂移 10% | 未解释为何上调 |

**修正建议**: 
- 要么删除精确比例，改用定性描述（"显著提升"）
- 要么给出明确的度量定义和计算方式

### 2.2 per-tool 频率上限与实际使用量不匹配

| 维度 | 数值 | 说明 |
|------|------|------|
| per-tool 上限之和 | 180/天 | search:50 + list:30 + detail:80 + profile:20 |
| 实际预估使用量 | 23-40/天 | 报告 §2.2 数据 |
| 留出安全边际 | 仅 10% | 远低于合理的 2-3x 余量 |

**修正建议**: 收紧 per-tool 上限至更贴近实际使用量，例如：
- search_feeds: 20/天（实际 10-15）
- list_feeds: 10/天（实际 2-3）
- get_feed_detail: 40/天（实际 10-20）
- user_profile: 10/天（实际 1-2）

## 三、遗漏清单

### 3.1 反例记录不充分（宪法 C5 违反）

**宪法要求**: 凡建议纳入非官方只读能力，必须"记录反例与失败信号（封号、限流、验证码、Cookies 失效）"。

**报告现状**: 
- 证据矩阵提到"仍有被检测为自动化的可能"
- 未系统性列出已知失败案例
- 未交叉验证 GitHub Issues 中是否有封号/限流报告

**修正建议**: 增加"已知失败信号清单"小节：
- GitHub Issues 中是否有封号/限流相关报告
- 社区中是否有使用该工具被风控的案例
- 若搜索不到反例，明确标注"未发现公开反例，但样本量不足以构成安全结论"

### 3.2 定期复核机制未纳入

**宪法要求**: 超过 6 个月未复核的平台规则或经验结论，不得作为强结论。

**报告现状**: 
- 实施路线图提到"季度复盘"
- 未将"宪法合规性复核"作为独立定期任务

**修正建议**: 在路线图中增加：
- 每 3 个月进行一次"小红书社区规范 + 反爬策略"复核
- 每次复核产出一份简要评估文档
- 若发现规则变化与现方案冲突，触发降级或停用

### 3.3 MCP Gateway 实现路径不清晰

**问题**: 报告提出 MCP Gateway 层作为"核心新增组件"，给出 YAML 配置，但没有说明：
- 是需要从零开发的中间件？
- 还是可以通过 MCPorter 现有能力实现？
- 开发量是多少？

**修正建议**: 明确标注 MCP Gateway 实现选项：
- 选项 A：MCPorter 已有 server 管理 + OpenClaw 配置（推荐 MVP）
- 选项 B：开发轻量 wrapper 脚本（预计 4-8h）
- 选项 C：开发完整中间件（预计 2-3 天）

## 五、结论与证据匹配

### 5.1 效率提升数字缺乏基准

**问题**: 报告多处引用"60-80% 效率提升"，但：
- 没有定义基准（时间？数据覆盖？决策质量？）
- 未解释为何比 Gemini 扫描的 50-70% 上调 10%

**修正建议**: 
- 删除精确比例，改用定性描述
- 或给出明确定义：如"人工搜索 30 分钟/天 → MCP 搜索 5 分钟/天 = 节约 83%"

### 5.2 成本计算存在乐观偏差

| 计算项 | 报告数据 | 问题 |
|--------|---------|------|
| 人工搜索节约 | ¥300/月 (6h × ¥50/h) | ¥50/h 是谁的成本？非现金节约 |
| 人工审核时间 | 未计入 | MCP 搜索后仍需人工审核筛选 |

**修正建议**: 
- 区分"现金成本节约"和"时间成本节约"
- 对于个人运营者，更有说服力的表达："每周节约 X 小时重复性操作"
