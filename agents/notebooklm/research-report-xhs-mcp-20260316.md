## 执行摘要

本报告基于宪法简报的核心约束——**"非官方写入自动化一律禁止，非官方只读能力可在低频、可审计、可随时停用前提下有限纳入"**——对 xiaohongshu-mcp 与现有 4 Agent 体系的最优集成方案进行深度研究。

### 核心结论

1. **推荐方案**: 方案 C+（增强型只读集成）— 在原方案 C 基础上，将 MCP 只读能力深度嵌入 xhs-radar 和 xhs-analyst 的工作流，同时为 xhs-social 提供评论样本采集能力。
2. **架构变更**: 新增 MCP Gateway 层，作为 4 Agent 与 xiaohongshu-mcp 之间的统一访问控制和审计中介。
3. **效率提升**: xhs-radar 效率预计提升 60-80%，xhs-analyst 数据获取效率提升 40-60%，xhs-social 评论分析效率提升 30-50%。
4. **写入功能**: 严格遵守宪法，所有写入操作禁止进入生产。但提供分级评估供晨星参考未来路径。
5. **成本变化**: MCP 服务本身零成本（自托管），但需增加 ¥50-100/月的监控和维护开销。

### 1.3 MCP Gateway 层设计

这是集成方案的**核心新增组件**，负责在 Agent 与 MCP 服务之间实施宪法约束：

```yaml
# mcp-gateway-config.yaml
gateway:
  whitelist:
    - check_login_status
    - get_login_qrcode
    - delete_cookies
    - search_feeds
    - list_feeds
    - get_feed_detail
    - user_profile
  
  blacklist:  # 硬编码，不可通过配置覆盖
    - publish_content
    - publish_with_video
    - post_comment_to_feed
    - reply_comment_in_feed
    - like_feed
    - favorite_feed
  
  rate_limits:
    global:
      max_requests_per_day: 200         # 宪法约束：单日 ≤ 200
      min_interval_seconds: 5           # 宪法约束：间隔 ≥ 5s
      max_requests_per_hour: 30         # 额外保护
    per_tool:
      search_feeds:
        max_per_day: 50
        min_interval_seconds: 10        # 搜索更保守
      list_feeds:
        max_per_day: 30
        min_interval_seconds: 15
      get_feed_detail:
        max_per_day: 80
        min_interval_seconds: 5
      user_profile:
        max_per_day: 20
        min_interval_seconds: 30        # 用户资料更敏感
  
  circuit_breaker:
    error_threshold: 3                  # 连续 3 次失败触发熔断
    reset_timeout_minutes: 30           # 30 分钟后自动重试
    captcha_detected: immediate_stop    # 验证码 → 立即停止
    login_anomaly: immediate_stop       # 登录异常 → 立即停止
  
  audit:
    log_all_requests: true
    log_path: ~/.openclaw/workspace/agents/xhs-mcp-audit/
    alert_channel: -5131273722          # 监控群
    sensitive_alerts:
      - rate_limit_reached
      - circuit_breaker_triggered
      - login_status_lost
      - captcha_detected
```

## 二、数据读取功能深度评估

### 2.1 功能逐项评估

#### search_feeds — 内容搜索

| 维度 | 评估 |
|------|------|
| **功能描述** | 按关键词搜索小红书笔记，支持筛选排序 |
| **参数** | `keyword`(必需), `sort_by`, `note_type`, `publish_time`, `search_scope`, `location` |
| **对接 Agent** | xhs-radar（主）|
| **替代方案** | 人工搜索 → 截图 → AI 分析；或第三方平台（千瓜数据） |
| **核心价值** | ⭐⭐⭐⭐⭐ — 热点追踪从"人工每日搜索 30 分钟"变为"自动搜索 5 分钟" |
| **风险评估** | 中 — 高频搜索可能触发反爬 |
| **推荐策略** | 每日 3-5 批次搜索，每批 3-5 个关键词，间隔 ≥ 10s |

**使用场景设计**:
```
xhs-radar 每日 06:00 热点扫描流程：
1. 搜索行业关键词（3-5 个） → search_feeds(keyword="xxx", sort_by="热度")
2. 搜索竞品品牌词（2-3 个） → search_feeds(keyword="竞品名")
3. 搜索趋势话题（2 个） → search_feeds(keyword="话题", publish_time="近7天")
4. 汇总 → 生成 daily-topics-{date}.md
预计总调用：10-15 次/天
```

#### list_feeds — 首页推荐列表

| 维度 | 评估 |
|------|------|
| **功能描述** | 获取首页推荐内容流 |
| **参数** | 无参数 |
| **对接 Agent** | xhs-radar |
| **替代方案** | 人工浏览首页 → 截图 |
| **核心价值** | ⭐⭐⭐ — 了解平台推荐趋势，但个性化推荐与账号权重相关，信息密度不如定向搜索 |
| **风险评估** | 低 — 模拟正常浏览行为 |
| **推荐策略** | 每日 2-3 次，作为搜索的补充 |

**使用场景设计**:
```
xhs-radar 推荐趋势采样：
1. 早间采样 06:00 → list_feeds() — 获取推荐流快照
2. 晚间采样 20:00 → list_feeds() — 对比推荐变化
3. 分析推荐趋势偏移 → 纳入选题参考
预计总调用：2-3 次/天
```

#### get_feed_detail — 帖子详情 + 评论

| 维度 | 评估 |
|------|------|
| **功能描述** | 获取指定帖子的完整内容、互动数据和评论列表 |
| **参数** | `feed_id`(必需), `xsec_token`(必需) |
| **对接 Agent** | xhs-social（评论获取）、xhs-analyst（数据采集） |
| **替代方案** | 人工截图评论区 → AI 用 image 工具提取 |
| **核心价值** | ⭐⭐⭐⭐⭐ — **最高价值工具**。直接获取结构化评论数据，替代"截图 → OCR → 分析"的低效链路 |
| **风险评估** | 中 — 需要 xsec_token，高频获取可能触发风控 |
| **推荐策略** | 优先获取自己发布内容的评论；竞品内容按需获取，单日 ≤ 20 篇 |

**使用场景设计**:
```
xhs-social 评论分析流程（取代人工截图）：
1. 获取自己最近发布帖子的评论 → get_feed_detail(feed_id, xsec_token)
2. AI 分析评论分类（问题/赞美/建议/负面）
3. 生成回复建议
4. 人工确认 + 手动回复
预计总调用：10-20 次/天

xhs-analyst 数据采集：
1. 获取自己历史帖子的互动数据 → get_feed_detail
2. 获取对标竞品爆款帖子的内容和数据
3. 生成数据对比分析
预计总调用：10-20 次/天
```

#### user_profile — 用户主页

| 维度 | 评估 |
|------|------|
| **功能描述** | 获取指定用户的主页信息 |
| **参数** | `user_id`(必需), `xsec_token`(必需) |
| **对接 Agent** | xhs-analyst |
| **替代方案** | 人工查看竞品主页 → 手动记录 |
| **核心价值** | ⭐⭐⭐ — 竞品账号监控有价值，但频率需求低 |
| **风险评估** | 中高 — 查看他人用户资料比搜索更敏感 |
| **推荐策略** | 每周 1-2 次批量更新竞品账号数据，不做实时监控 |

**使用场景设计**:
```
xhs-analyst 竞品账号追踪（周度）：
1. 获取 5-10 个竞品账号主页数据 → user_profile
2. 记录粉丝增长、发帖频率、互动变化
3. 生成竞品周报
预计总调用：5-10 次/周
```

## 四、写入功能分级评估

> **宪法立场**: 所有写入类自动化在生产环境中禁止。以下评估是为了给晨星提供**技术可行性与风险认知**，而非推荐使用。

### 4.1 自动发帖 (`publish_content` / `publish_with_video`)

| 维度 | 评估 |
|------|------|
| **技术可行性** | ✅ 可实现，支持图文和视频，含定时发布 |
| **风险等级** | 🔴 极高 |
| **封号概率** | 30-90 天内高概率触发风控（Gemini 扫描评估） |
| **法律风险** | 中 — AIGC 标注合规可控，但非官方 API 使用违反平台协议 |

**分级建议**:

| 条件 | 可考虑程度 | 前提 |
|------|-----------|------|
| 测试账号 + 非商用 | ⚠️ 可探索 | 非主号、可接受封号风险、学习目的 |
| 人工审核 → MCP 执行发布 | ❌ 不建议 | 即使人工审核内容，自动化发布本身就是风控触发点 |
| 主运营账号 | ❌ 绝对禁止 | 封号代价不可承受 |

**如果未来要考虑（非当前推荐）**:
- 必须在**独立测试账号**上运行 ≥ 30 天无封号
- 必须实现**完整人工审核门控**（AI 生成 → 人工审核 → 人工确认 → MCP 执行发布）
- 必须支持**一键回退到纯人工发布**
- 必须记录每次发布的完整审计日志
- 发布频率 ≤ 2 篇/天，与正常人工发布频率一致

### 4.2 自动点赞/收藏 (`like_feed` / `favorite_feed`)

| 维度 | 评估 |
|------|------|
| **技术可行性** | ✅ 简单易实现 |
| **风险等级** | 🔴 高 |
| **封号概率** | 取决于频率，低频（<10/天）短期风险低，但长期不确定 |
| **法律风险** | 高 — 构成流量造假，可能涉及《反不正当竞争法》 |

**分级建议**:

| 场景 | 风险 | 建议 |
|------|------|------|
| 自动点赞竞品内容（假互动） | 极高 | ❌ 绝对禁止 |
| 自动收藏自己的内容 | 中 | ❌ 无意义且触发风控 |
| 冷启动养号期批量点赞 | 高 | ❌ 禁止，养号行为用人工 |
| 低频互惠点赞（<5/天） | 中 | ⚠️ 不建议，人工操作成本极低 |

**频率限制建议（即使未来考虑）**:
- ≤ 5 次/天
- 随机间隔 30s-5min
- 仅在浏览行为之后（模拟"看完点赞"）
- 绝不批量、绝不定时

### 4.3 自动回复 (`post_comment_to_feed` / `reply_comment_in_feed`)

| 维度 | 评估 |
|------|------|
| **技术可行性** | ✅ 支持精确回复指定评论 |
| **风险等级** | 🔴 高 |
| **封号概率** | 高 — 自动评论是平台反垃圾重点打击项 |
| **品牌风险** | 高 — AI 回复失误可能造成 PR 危机 |

**分级建议**:

| 场景 | 风险 | 建议 |
|------|------|------|
| 自动回复自己帖子下的评论 | 高 | ❌ 不建议生产使用 |
| 自动在竞品帖子下评论 | 极高 | ❌ 绝对禁止（垃圾信息 + 引流嫌疑） |
| AI 生成 → 人工确认 → MCP 发送 | 中高 | ⚠️ 不推荐，但技术上风险比纯自动低 |
| AI 生成 → 人工手动粘贴发送 | 低 | ✅ **推荐方案**（现有方案已涵盖） |

**核心理由**: 
- 评论回复是**高品牌风险**操作，AI 理解语境/讽刺/暗语的能力有限
- 即使 AI 生成内容完全正确，自动化发送本身也是风控触发点
- 人工粘贴发送的额外成本极低（每条 <10 秒），但获得的安全边际极大

### 4.4 写入功能总结

| 写入操作 | 技术可行 | 生产推荐 | 替代方案 | 备注 |
|---------|---------|---------|---------|------|
| 自动发帖 | ✅ | ❌ 禁止 | AI 生成 + 人工发布 | 封号风险不可接受 |
| 自动点赞 | ✅ | ❌ 禁止 | 人工点赞 | 法律+流量造假风险 |
| 自动收藏 | ✅ | ❌ 禁止 | 人工收藏 | 同上 |

## 五、风险缓解清单

### 5.1 Cookies 管理

| 风险 | 缓解措施 | 优先级 |
|------|---------|--------|
| Cookies 过期导致功能中断 | 每日 `check_login_status` 健康检查；过期时通知人工重新扫码 | 🔴 P0 |
| Cookies 泄露导致账号安全 | Cookies 文件权限 600，不纳入版本控制，不跨设备传输 | 🔴 P0 |
| 扫码登录窗口期短 | 设置 cron 提醒，工作时间内尽快完成扫码 | 🟡 P1 |
| 多设备登录冲突 | 仅在 MCP 部署机器上维护登录态，不与移动端冲突 | 🟡 P1 |

**自动巡检脚本设计**:
```bash
# 每日 05:30 执行（在 radar 启动前）
check_login_status → 
  if logged_in: 
    log "✅ 登录正常" 
  else: 
    alert "⚠️ Cookies 失效，请在 06:00 前扫码重新登录"
    → 发送到珊瑚群 + 监控群
    → MCP Gateway 进入降级模式（跳过 MCP 调用，用备用数据源）
```

### 5.2 频率限制

| 层级 | 限制 | 实现方式 |
|------|------|---------|
| 全局 | ≤ 200 次/天 | MCP Gateway 计数器 |
| 全局 | ≥ 5s 间隔 | MCP Gateway 节流阀 |
| 逐工具 | 见 §3.1 矩阵 | MCP Gateway 分工具计数 |
| 逐小时 | ≤ 30 次/小时 | 防止短时集中调用 |
| 随机化 | 间隔 = 基础间隔 × (1 + random(0, 0.5)) | 避免固定节奏被检测 |

### 5.3 降级策略

```
MCP 降级瀑布：
Level 0: MCP 正常 ──────────── 全功能使用
Level 1: MCP 限流 ──────────── 降低调用频率 50%
Level 2: MCP 熔断 ──────────── 停用 MCP，使用备用数据源
Level 3: Cookies 失效 ────────── 停用 MCP，等待人工重新登录
Level 4: 账号异常 ──────────── 停用一切自动化，仅人工操作

每个级别的回退方案：
┌─────────────┬──────────────────────────────────┐
│ Agent       │ 降级回退                           │
├─────────────┼──────────────────────────────────┤
│ xhs-radar   │ web_search + 第三方平台 + 人工搜索  │
│ xhs-social  │ 人工截图 + image 工具分析           │
│ xhs-analyst │ 人工导出创作者后台 CSV + AI 分析    │
│ xhs-writer  │ 无影响（不使用 MCP）               │
└─────────────┴──────────────────────────────────┘
```

| MCP 服务存活 | `localhost:18060` 不可达 | 监控群 | 重启服务或降级 |

## 十、关键建议与开放问题

### 10.1 给晨星的核心建议

1. **立即可做**: 部署 xiaohongshu-mcp + MCP Gateway，仅启用只读功能，与 xhs-radar 对接。这是**低风险、高回报**的增量升级。
