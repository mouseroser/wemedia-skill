## 1. 执行摘要 (Executive Summary)

### 核心发现

OpenClaw System Prompt 采用了一个高度复杂且精密的 9 层叠加架构，自下而上分为框架生成层（L1-L6）、用户静态控制层（L7）、用户动态注入层（L8）以及实时流量层（L9）。该架构通过 L7 的工作区文件和 L8 的可编程注入器，赋予了用户极大的灵活性，支持复杂的智能体状态注入与实时编排。

### 架构风险概览

尽管设计具有高度可定制性，但该架构面临三个核心风险：

1. **上下文膨胀 (Context Bloat)**：随着用户用例的增加，L7 静态文件和 L8 动态注入信息过多，极易逼近或超出模型上下文窗口限制，引发由于 `bootstrapMaxChars` 机制不透明导致的截断风险。

2. **指令冲突与执行不可见性**：L7 静态规则与 L8 动态脚本之间缺乏明确的冲突覆盖说明；同时，L1-L6 完全由底层框架生成（黑盒特性），增加了用户排查模型非预期行为的难度。

3. **安全边界模糊**：L9 的外部实时流量和 L8 的动态脚本可能成为外部恶意指令注入 (Prompt Injection) 的载体，对 L1-L6 核心框架协议的越权篡改防护机制尚存疑虑。

## 2. 架构层级分析 (Layer-by-Layer Analysis)

根据原始架构图与宪法简报，9 层架构的功能与潜在问题拆解如下：

### 框架底层黑盒区 (L1-L6)

完全由框架生成，用户不可直接控制。

#### L1 OpenClaw Core
- **功能**: 提供基础的 Agent 身份声明、当前时间/环境、XML 工具调用规范
- **核心**: **安全边界 (Safety Boundaries)** - 最重要的底层约束
- **特点**: 用户不可修改的基础协议层

#### L2 Tool Definitions
- **功能**: "瑞士军刀"层，包含 read, write, list 等基础工具的 JSON Schema 定义
- **作用**: 定义智能体可用的工具接口规范

#### L3 Skills Registry
- **功能**: 采用自动目录扫描机制，加载自定义技能的系统提示词片段
- **潜在问题**: 每次构建时的自动扫描可能带来额外的性能与延迟开销

#### L4 Model Aliases
- **功能**: 提供模型快捷别名映射（如 gln-5, opus, sonnet）并支持模型切换
- **作用**: 简化模型调用，支持动态切换

#### L5 Protocol Specs
- **功能**: 定义静默回复、心跳机制、回复标签等底层通信流量规则
- **作用**: 规范智能体与外部系统的交互协议

#### L6 Runtime Info
- **功能**: 实时状态仪表盘，注入主机、操作系统及模型等状态
- **作用**: 提供运行时环境感知能力

**核心挑战**: 这六层的黑盒特性使得调试和溯源变得困难，用户必须承认其对整体安全和功能的前提约束。

### 用户静态控制层 (L7)

#### L7 Workspace Files
- **内容**: 包含 `IDENTITY.md`（身份/目标）、`AGENTS.md`（规则）和 `MEMORY.md` 等直接编辑的静态配置文件
- **特点**: 用户完全可控，直接编辑
- **潜在问题**: 静态文件极易变得冗长并产生指令重复与冲突，导致上下文膨胀

### 用户动态注入层 (L8)

#### L8 Bootstrap Hook
- **功能**: 可编程注入器，包含 `agent:bootstrap`、`bootstrap-extra-files`、`before_prompt_build` 等动态脚本机制
- **容量控制**: 通过 `bootstrapMaxChars` 进行容量管理
- **潜在问题**: 针对不同场景正确选择注入方式困难，存在被滥用的风险

### 实时流量接入层 (L9)

#### L9 Inbound Context
- **内容**: 包含消息元信息、发送者信息及实时对话历史
- **特点**: 直接接触外部输入
- **潜在问题**: 作为直接接触外部输入的一层，缺乏防范恶意指令注入的明确机制描述

## 3. 关键问题回应 (Key Issues Resolved)

### 冲突与优先级机制解析（L7 vs L8）

L7 提供基础的静态设定，而 L8 提供动态的运行时注入。从架构设计的钩子命名（如 `before_prompt_build`）可以看出，L8 的动态脚本在提示词最终组装阶段执行。

**结论**: 根据就近原则与时间线顺序，L8 的动态上下文在逻辑上晚于 L7，因此 L8 在实际构建顺序中拥有事实上的更高优先级，能够临时覆盖或修正 L7 中的冲突指令。

### 上下文容量与截断策略评估（bootstrapMaxChars 机制）

当 9 层架构叠加海量前置上下文时，Token 容量管理成为巨大挑战。L8 层专门引入了 `bootstrapMaxChars Config` 用于控制注入大小。

**分析**: 如果该机制采用硬性报错或随机丢弃，将导致整个 L9（实时流量）被破坏或模型崩溃。

**合理预期**: 系统采用对历史对话 (L9) 或外挂文档 (L8) 的智能截断，以保护 L1 核心协议不被破坏，但这仍需具体代码实现的进一步证实。

### L8 权限边界与安全风险分析

这是一个核心隐患。架构图显示 L1 包含了"安全边界" (Safety Boundaries)。根据宪法简报的判定标准，绝不能在缺乏具体证据的情况下凭空推断 L8 的权限可以随意覆盖 L1-L6。

**结论**: 
- L8 的脚本执行环境理应受到严格的沙箱限制
- 其越权修改能力仅局限于重塑 L7 到 L9 的用户层上下文
- 无权篡改或删除 L1-L6 框架底层生成的内置系统提示词
- L8 和 L9 确实亟需引入显式的 Prompt Injection 防范机制

## 4. 假设验证结论 (Hypothesis Verification)

针对 Gemini 扫描报告中提出的 4 个假设，逐一验证如下：

### 假设 1：L8 覆写优先级

**假设内容**: L8 位于 L7 之后或优先级更高，可临时覆盖 L7。

**验证结论**: ✅ **成立**

根据动态注入器（如 `before_prompt_build`）的特性，它在提示词构建的最后环节动态组装上下文，时间顺序晚于 L7 静态文件加载，具备覆盖低优静态设定的能力。

### 假设 2：L8 越权修改能力

**假设内容**: L8 可拦截、修改或删除 L1-L6 内置提示词。

**验证结论**: ❌ **证伪**

宪法简报明确禁止"在没有具体场景和验证支撑的情况下，凭空推断 L8 的权限可以随意覆盖 L1-L6"。基于 L1 必须维持的基础"Safety Boundaries"，L8 受到底层约束，不具备破坏黑盒核心层的最高权限。

### 假设 3：截断策略

**假设内容**: 触发 bootstrapMaxChars 时，采用智能截断而非随机丢弃。

**验证结论**: ⚠️ **部分验证 / 逻辑上成立**

为了防止覆盖实时流量 (L9) 或导致整体崩溃，系统必须依靠对非核心内容（如 L9 历史对话或 L8 长文档）的智能截断来保障 L1-L6 核心框架完整性。不过，具体截断算法细节仍属盲区。

### 假设 4：动态模型切换

**假设内容**: L4 模型切换不丢失 L5-L9 累积状态。

**验证结论**: ⚠️ **待进一步验证**

L4 (Model Aliases) 确实旨在支持模型快捷切换，但在多智能体协作与复杂工作流中，如何完美共享和同步跨模型的 L5 协议状态与 L7 的 `MEMORY.md`，目前框架图中并未给出深入的同步机制说明。

## 5. 优化与加固建议 (Recommendations)

结合架构全景图的官方优化建议及识别出的风险，提出以下加固方案：

### 1. 针对上下文膨胀 (L7 优化)

**问题**: L7 静态文件容易变得冗长，导致上下文膨胀和指令冲突。

**建议**:
- 严格执行"Refine, use checklists, avoid repeats"原则
- 定期重构 `IDENTITY.md` 和 `AGENTS.md`
- 使用精简的清单体替代长篇大论
- 避免与 L8 动态内容发生语义重复

### 2. 针对动态注入混乱 (L8 规范化)

**问题**: L8 提供多种注入方式，容易被滥用或误用。

**建议**:

#### 场景化分级注入
- **简单的静态文件补充**: 仅使用 `bootstrap-extra-files`
- **复杂的初始化逻辑**: 交由 `agent:bootstrap` 处理
- **强实时控制需求**: 才动用高消耗的 `before_prompt_build` 钩子

#### 引入监控面板
- 量化每次 Prompt 构建时 L3 自动扫描和 L8 脚本注入产生的性能与延迟开销
- 提供可视化的上下文使用情况

### 3. 针对安全防护与权限边界加固 (L1, L8, L9)

**问题**: 安全边界模糊，存在 Prompt Injection 风险。

**建议**:

#### 固化黑盒防篡改
- 在代码层面强制隔离
- 确保 L1-L6 的提示词字符串对 L8 脚本完全只读
- 阻断越权修改路径

#### 抗注入 (Prompt Injection) 过滤
- 在 L9 接收实时流量和 L8 挂载外部脚本之前，增加专门的安全清洗层
- 声明忽略任何试图重写 L1-L2（如身份声明与安全边界）的指令
- 实施输入验证和内容过滤

#### 明确截断策略透明度
官方应公布 `bootstrapMaxChars` 的确切智能截断顺序，推荐实施：

**优先级顺序**:
1. 绝对保留 L1-L6（核心框架）
2. 保留 L7 核心身份
3. 截断 L8 外挂文档
4. 滑动截断 L9 早期历史

---

## 总结

OpenClaw System Prompt 的 9 层架构设计精巧，提供了强大的灵活性和可扩展性。然而，这种复杂性也带来了上下文管理、安全防护和调试可见性方面的挑战。通过实施上述优化和加固建议，可以显著提升系统的稳定性、安全性和可维护性。

**关键要点**:
- L8 在构建顺序上晚于 L7，具有更高的事实优先级
- L8 不应具备修改 L1-L6 核心框架的权限
- 需要明确的截断策略和 Prompt Injection 防护机制
- 定期优化 L7 静态文件，规范化 L8 动态注入使用
