## 1. 范围（Scope）
### 1.1 本次允许研究与纳入的边界
- 仅评估 **单账号、中文市场、小规模运营** 下的 OpenClaw 辅助运营方案。
- 允许将 xiaohongshu-mcp 作为 **情报采集/公开内容读取层**，服务于选题、竞品观察、评论样本分析、账号洞察。
- 发布、评论回复、互动动作仍以 **人工在小红书官方客户端内执行** 为唯一生产路径。

### 1.2 明确不包含
- 不包含“官方内容发布 MCP”假设；**当前不存在面向个人内容运营的官方小红书 MCP/API**。
- 不包含自动发帖、自动评论/回复、自动点赞/收藏、自动引流、矩阵号批量运营。
- 不包含绕过风控、规避验证码、代理池/养号、批量抓取或用户隐私采集。

## 2. 定义（Definitions）
- **官方合规**：基于小红书官方开放平台、创作者后台、人工手动操作形成的能力边界。当前官方开放平台以电商 API 为主，**不覆盖个人内容发布自动化**。
- **非官方但可控风险**：基于浏览器模拟 + Cookies 持久化实现的 xiaohongshu-mcp **只读**能力；前提是单账号、低频、最小必要、全程留痕、可一键停用。
- **写入类操作**：任何改变平台状态的动作，包括 `publish_*`、评论/回复、点赞、收藏。**只要通过 MCP 执行，即视为自动化写入**。
- **人工确认**：AI 可以生成草稿、标题、回复建议；但最终发布/回复/互动必须由人工在官方客户端完成，而非通过 MCP 调用完成。
- **引流**：将用户导向微信、淘宝、私域或其他站外渠道。此类动作在小红书属于高敏感区，必须保守处理。

## 3. 判定标准（Criteria）
### 3.1 风险分级与准入规则
| 等级 | 类别 | 典型能力 | 决策 |
|---|---|---|---|
| R0 | 官方合规 | 创作者后台人工导出、人工浏览、官方电商 API | 允许 |
| R1 | 非官方只读 | `search_feeds` / `list_feeds` / `get_feed_detail` / `user_profile` / 登录状态维护 | **条件允许** |
| R2 | AI 建议但人工执行 | 发帖文案、标题、评论回复建议、CTA 草稿 | 允许，但必须人工落地 |
| R3 | 非官方写入自动化 | `publish_content` / `publish_with_video` / `post_comment_to_feed` / `reply_comment_in_feed` / `like_feed` / `favorite_feed` | **生产禁止** |

### 3.2 xiaohongshu-mcp 功能白名单 / 黑名单
**白名单（可纳入）**
- 账号登录辅助：`check_login_status`、`get_login_qrcode`、`delete_cookies`
- 只读采集：`search_feeds`、`list_feeds`、`get_feed_detail`、`user_profile`

**黑名单（不得纳入生产运营）**
- 内容发布：`publish_content`、`publish_with_video`、`schedule_at`
- 互动写入：`post_comment_to_feed`、`reply_comment_in_feed`、`like_feed`、`favorite_feed`
- 任何基于上述能力扩展出的自动养号、自动引流、批量互动

### 3.3 只读能力的最低控制条件
- 必须 **单账号、单会话、默认串行请求**。
- 默认请求间隔 **≥ 5 秒**；初始阶段单日总调用量 **≤ 200**，超限前必须先做风控复核。
- 只采集公开页面所需最小数据；不得沉淀用户手机号、微信号等个人信息。
- 出现验证码、异常登录提醒、限流提示、账号安全提醒时，**立即停用 MCP**。

### 3.4 对原始需求的现实裁决
| 原始需求 | 技术上是否可做 | 风险判断 | 本次裁决 |
|---|---|---|---|
| 自动发帖 | 可通过非官方工具部分实现 | 高封号风险（扫描给出 30-90 天高风险区间） | 不纳入 |
| 自动点赞/收藏 | 可实现 | 高风控 + 流量造假风险 | 禁止 |
| 自动回复评论 | 可实现 | 高风控 + 品牌/垃圾信息风险 | 不纳入；改为 AI 建议 + 人工手动回复 |
| 自动引流 | 可通过文案/评论变相实现 | 平台高敏感，违规概率高 | 禁止自动化；仅允许人工合规弱引导 |

## 4. 禁止项（Red Lines）
1. 不得把 xiaohongshu-mcp 描述为“小红书官方 MCP”或“官方认可集成”。
2. 不得在生产账号上启用任何 MCP 写入类动作。
3. 不得以“作者声称稳定运行一年多”作为上线充分依据；该信息仅算**单一逸闻证据**。
4. 不得设计任何规避风控、绕过验证码、代理轮换、矩阵放大、批量采集方案。
5. 不得通过评论、私信、点赞、收藏自动化进行站外导流或流量操纵。
6. 一旦出现封禁、限流、异常登录、验证码升级、社区规则更新与现方案冲突，必须立即停止相关模块并回退到纯人工流程。

## 5. 证据门槛（Evidence Threshold）
- **凡声称“官方合规”**，必须引用小红书官方开放平台/社区规范原文；不得用 README、博客或 issue 替代。
- **凡建议纳入非官方只读能力**，至少满足：
  1) 项目代码/README 能证明功能存在；
  2) 至少 1 份外部反馈（issue、使用案例、测试记录）证明其可运行；
  3) 必须记录反例与失败信号（封号、限流、验证码、Cookies 失效）。
- 超过 **6 个月** 未复核的平台规则或经验结论，不得作为强结论。
- 对“风险可控”的结论，必须同时写明：使用条件、停用触发器、回退路径。
