## 1. 执行摘要

### 架构概览

OpenClaw System Prompt 采用 9 层叠加架构，自下而上分为：

- **L1-L6**: 框架生成层（黑盒，用户不可控）
- **L7**: 用户静态控制层（工作区文件）
- **L8**: 用户动态注入层（可编程钩子）
- **L9**: 实时流量层（外部输入）

该架构通过 L7 的工作区文件和 L8 的可编程注入器，赋予用户极大的灵活性，支持复杂的智能体状态注入与实时编排。

### 三大核心风险

#### 🔴 风险 1: 上下文膨胀 (Context Bloat) - 高风险

**问题**: 
- L7 静态文件和 L8 动态注入信息过多
- 极易逼近或超出模型上下文窗口限制
- `bootstrapMaxChars` 截断机制不透明

**影响**: 
- 关键指令被截断
- 模型行为不可预测
- 调试困难

**缓解方案**:
- 定期重构 L7 文件，使用清单体替代长文
- 规范化 L8 注入使用，避免滥用
- 公开截断策略，提供监控面板

#### 🟡 风险 2: 指令冲突与执行不可见性 - 中风险

**问题**:
- L7 静态规则与 L8 动态脚本缺乏明确的冲突覆盖说明
- L1-L6 完全黑盒，用户难以排查非预期行为

**影响**:
- 指令优先级不明确
- 调试成本高
- 用户体验差

**缓解方案**:
- 明确 L7/L8 优先级规则（L8 > L7）
- 提供 L1-L6 的文档说明（不暴露源码）
- 增加调试日志和可观测性

#### 🔴 风险 3: 安全边界模糊 - 高风险

**问题**:
- L9 外部实时流量和 L8 动态脚本可能成为 Prompt Injection 载体
- L1-L6 核心框架的越权篡改防护机制不明确

**影响**:
- 恶意指令注入
- 安全边界被绕过
- 系统完整性受损

**缓解方案**:
- 在 L8/L9 前增加安全清洗层
- 固化 L1-L6 防篡改机制
- 实施输入验证和内容过滤

## 2. 架构层级分析

### 2.1 框架底层黑盒区 (L1-L6)

完全由框架生成，用户不可直接控制。

#### L1: OpenClaw Core - 基础协议层
- **功能**: Agent 身份声明、时间/环境、XML 工具调用规范
- **核心**: **安全边界 (Safety Boundaries)** - 最重要的底层约束
- **特点**: 用户不可修改的基础协议

#### L2: Tool Definitions - 工具接口层
- **功能**: 基础工具的 JSON Schema 定义（read, write, list 等）
- **作用**: 定义智能体可用的工具接口规范

#### L3: Skills Registry - 技能注册层
- **功能**: 自动目录扫描，加载自定义技能的系统提示词片段
- **潜在问题**: 自动扫描可能带来性能与延迟开销

#### L4: Model Aliases - 模型切换层
- **功能**: 模型快捷别名映射（gln-5, opus, sonnet）
- **作用**: 简化模型调用，支持动态切换

#### L5: Protocol Specs - 通信协议层
- **功能**: 静默回复、心跳机制、回复标签等底层通信规则
- **作用**: 规范智能体与外部系统的交互协议

#### L6: Runtime Info - 运行时状态层
- **功能**: 实时状态仪表盘（主机、操作系统、模型等）
- **作用**: 提供运行时环境感知能力

**核心挑战**: 黑盒特性使得调试和溯源困难，用户必须信任底层约束。

### 2.2 用户静态控制层 (L7)

#### L7: Workspace Files - 工作区文件层
- **内容**: `IDENTITY.md`（身份/目标）、`AGENTS.md`（规则）、`MEMORY.md`（记忆）等
- **特点**: 用户完全可控，直接编辑
- **潜在问题**: 静态文件易变冗长，导致指令重复与冲突

### 2.3 用户动态注入层 (L8)

#### L8: Bootstrap Hook - 可编程注入层
- **功能**: `agent:bootstrap`、`bootstrap-extra-files`、`before_prompt_build` 等动态脚本机制
- **容量控制**: 通过 `bootstrapMaxChars` 进行容量管理
- **潜在问题**: 注入方式选择困难，存在滥用风险

### 2.4 实时流量接入层 (L9)

#### L9: Inbound Context - 实时流量层
- **内容**: 消息元信息、发送者信息、实时对话历史
- **特点**: 直接接触外部输入
- **潜在问题**: 缺乏防范恶意指令注入的明确机制

## 3. 关键问题回应

### 3.1 冲突与优先级机制（L7 vs L8）

**问题**: L7 静态文件与 L8 动态脚本发生冲突时，谁优先？

**分析**: 
- L7 提供基础静态设定
- L8 提供动态运行时注入
- 从钩子命名（`before_prompt_build`）可知，L8 在提示词最终组装阶段执行

**结论**: 
✅ **L8 优先级更高**

根据就近原则与时间线顺序，L8 的动态上下文在逻辑上晚于 L7，因此在实际构建顺序中拥有事实上的更高优先级，能够临时覆盖或修正 L7 中的冲突指令。

### 3.2 上下文容量与截断策略（bootstrapMaxChars）

**问题**: 当 9 层架构叠加海量前置上下文时，如何管理 Token 容量？

**分析**: 
- L8 引入 `bootstrapMaxChars` 控制注入大小
- 如果采用硬性报错或随机丢弃，将导致 L9（实时流量）被破坏或模型崩溃

**合理预期**: 
⚠️ **智能截断，保护核心层**

系统应采用对历史对话 (L9) 或外挂文档 (L8) 的智能截断，以保护 L1 核心协议不被破坏。

**推荐截断优先级**:
1. **绝对保留**: L1-L6（核心框架）
2. **优先保留**: L7 核心身份
3. **可截断**: L8 外挂文档
4. **滑动截断**: L9 早期历史

### 3.3 L8 权限边界与安全风险

**问题**: L8 是否可以修改或删除 L1-L6 的内置提示词？

**分析**: 
- 架构图显示 L1 包含"安全边界" (Safety Boundaries)
- 宪法简报明确禁止"在没有具体场景和验证支撑的情况下，凭空推断 L8 的权限可以随意覆盖 L1-L6"

**结论**: 
❌ **L8 不应具备越权修改能力**

- L8 的脚本执行环境应受到严格的沙箱限制
- 其越权修改能力仅局限于重塑 L7 到 L9 的用户层上下文
- 无权篡改或删除 L1-L6 框架底层生成的内置系统提示词
- L8 和 L9 亟需引入显式的 Prompt Injection 防范机制

**技术实现建议**:
- 在代码层面强制隔离
- 确保 L1-L6 的提示词字符串对 L8 脚本完全只读
- 阻断越权修改路径

## 4. 假设验证结论

针对 Gemini 扫描报告中提出的 4 个假设，逐一验证：

### 假设 1: L8 覆写优先级
**假设**: L8 位于 L7 之后或优先级更高，可临时覆盖 L7。

**验证结论**: ✅ **成立**

根据动态注入器（如 `before_prompt_build`）的特性，它在提示词构建的最后环节动态组装上下文，时间顺序晚于 L7 静态文件加载，具备覆盖低优先级静态设定的能力。

### 假设 2: L8 越权修改能力
**假设**: L8 可拦截、修改或删除 L1-L6 内置提示词。

**验证结论**: ❌ **证伪**

宪法简报明确禁止凭空推断 L8 的越权能力。基于 L1 必须维持的基础"Safety Boundaries"，L8 受到底层约束，不具备破坏黑盒核心层的最高权限。

### 假设 3: 截断策略
**假设**: 触发 bootstrapMaxChars 时，采用智能截断而非随机丢弃。

**验证结论**: ⚠️ **逻辑上成立，需代码验证**

为了防止覆盖实时流量 (L9) 或导致整体崩溃，系统必须依靠对非核心内容（如 L9 历史对话或 L8 长文档）的智能截断来保障 L1-L6 核心框架完整性。具体截断算法细节仍属盲区。

### 假设 4: 动态模型切换
**假设**: L4 模型切换不丢失 L5-L9 累积状态。

**验证结论**: ⚠️ **待进一步验证**

L4 (Model Aliases) 确实旨在支持模型快捷切换，但在多智能体协作与复杂工作流中，如何完美共享和同步跨模型的 L5 协议状态与 L7 的 `MEMORY.md`，目前框架图中并未给出深入的同步机制说明。

## 5. 优化与加固建议

### 5.1 针对上下文膨胀 (L7 优化)

**问题**: L7 静态文件容易变得冗长，导致上下文膨胀和指令冲突。

**建议**:

#### 精简原则
- 严格执行"Refine, use checklists, avoid repeats"原则
- 定期重构 `IDENTITY.md` 和 `AGENTS.md`
- 使用精简的清单体替代长篇大论
- 避免与 L8 动态内容发生语义重复

#### 具体措施
- 每月审查 L7 文件，删除过时内容
- 使用模板和引用，避免重复定义
- 将详细说明移至外部文档，L7 只保留核心指令

### 5.2 针对动态注入混乱 (L8 规范化)

**问题**: L8 提供多种注入方式，容易被滥用或误用。

**建议**:

#### 场景化分级注入
- **简单的静态文件补充**: 仅使用 `bootstrap-extra-files`
- **复杂的初始化逻辑**: 交由 `agent:bootstrap` 处理
- **强实时控制需求**: 才动用高消耗的 `before_prompt_build` 钩子

#### 引入监控面板
- 量化每次 Prompt 构建时 L3 自动扫描和 L8 脚本注入产生的性能与延迟开销
- 提供可视化的上下文使用情况
- 设置告警阈值，防止超限

#### 使用指南
- 为每种注入方式提供清晰的使用场景说明
- 提供最佳实践示例
- 建立代码审查机制

### 5.3 针对安全防护与权限边界加固 (L1, L8, L9)

**问题**: 安全边界模糊，存在 Prompt Injection 风险。

**建议**:

#### 固化黑盒防篡改
- 在代码层面强制隔离
- 确保 L1-L6 的提示词字符串对 L8 脚本完全只读
- 阻断越权修改路径
- 实施完整性校验

#### 抗注入 (Prompt Injection) 过滤
在 L9 接收实时流量和 L8 挂载外部脚本之前，增加专门的安全清洗层：

**过滤规则**:
- 检测并忽略任何试图重写 L1-L2（如身份声明与安全边界）的指令
- 识别常见的 Prompt Injection 模式
- 实施输入验证和内容过滤
- 记录可疑输入，供审计使用

**具体措施**:
- 使用正则表达式或 NLP 模型检测恶意模式
- 建立黑名单和白名单机制
- 实施速率限制，防止暴力攻击

#### 明确截断策略透明度
官方应公布 `bootstrapMaxChars` 的确切智能截断顺序：

**推荐优先级**:
1. **绝对保留**: L1-L6（核心框架）
2. **优先保留**: L7 核心身份
3. **可截断**: L8 外挂文档
4. **滑动截断**: L9 早期历史

**透明度要求**:
- 提供截断日志
- 显示当前上下文使用情况
- 在截断发生时发出警告

## 6. 总结

### 核心发现

OpenClaw System Prompt 的 9 层架构设计精巧，提供了强大的灵活性和可扩展性。然而，这种复杂性也带来了上下文管理、安全防护和调试可见性方面的挑战。

### 关键要点

1. **优先级明确**: L8 在构建顺序上晚于 L7，具有更高的事实优先级
2. **权限边界**: L8 不应具备修改 L1-L6 核心框架的权限
3. **安全防护**: 需要明确的截断策略和 Prompt Injection 防护机制
4. **持续优化**: 定期优化 L7 静态文件，规范化 L8 动态注入使用

### 实施路径

**短期（1-2 周）**:
- 公开截断策略文档
- 增加 L8/L9 输入验证
- 提供 L7 优化指南

**中期（1-2 月）**:
- 实施监控面板
- 固化 L1-L6 防篡改机制
- 建立 L8 使用规范

**长期（3-6 月）**:
- 完善安全清洗层
- 提供调试工具
- 建立最佳实践库

### 风险评估

- **上下文膨胀**: 🔴 高风险 → 需立即优化
- **指令冲突**: 🟡 中风险 → 需明确规则
- **安全边界**: 🔴 高风险 → 需加固防护

通过实施上述优化和加固建议，可以显著提升系统的稳定性、安全性和可维护性。
