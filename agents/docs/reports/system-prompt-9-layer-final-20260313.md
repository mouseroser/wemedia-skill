## 1. 执行摘要

### 1.1 架构概览

OpenClaw System Prompt 采用精密的 **9 层叠加架构**，自下而上分为：

| 层级 | 名称 | 性质 | 控制权 |
|------|------|------|--------|
| L1 | OpenClaw Core | 框架生成层 | 黑盒 |
| L2 | Tool Definitions | 框架生成层 | 黑盒 |
| L3 | Skills Registry | 框架生成层 | 黑盒 |
| L4 | Model Aliases | 框架生成层 | 黑盒 |
| L5 | Protocol Specs | 框架生成层 | 黑盒 |
| L6 | Runtime Info | 框架生成层 | 黑盒 |
| L7 | Workspace Files | 用户静态层 | 可控 |
| L8 | Bootstrap Hook | 用户动态层 | 可控 |
| L9 | Inbound Context | 实时流量层 | 外部输入 |

### 1.2 核心发现

该架构通过 L7 工作区文件和 L8 可编程注入器，赋予用户极大的灵活性，支持复杂的智能体状态注入与实时编排。然而，这种复杂性也带来了三大核心挑战。

### 1.3 三大核心风险

| 风险 | 等级 | 描述 |
|------|------|------|
| **上下文膨胀** | 🔴 高 | L7/L8 信息过多，`bootstrapMaxChars` 截断机制不透明 |
| **指令冲突** | 🟡 中 | L7/L8 优先级不明确，L1-L6 黑盒调试困难 |
| **安全边界模糊** | 🔴 高 | L9/L8 缺乏 Prompt Injection 防范机制 |

## 2. 完整架构分析

### 2.1 框架底层黑盒区 (L1-L6)

完全由框架生成，用户不可直接控制。

#### L1: OpenClaw Core - 基础协议层
- **功能**: Agent 身份声明、时间/环境信息、XML 工具调用规范
- **核心**: **安全边界 (Safety Boundaries)** - 最重要的底层约束
- **特点**: 用户不可修改的基础协议

#### L2: Tool Definitions - 工具接口层
- **功能**: 基础工具的 JSON Schema 定义（read, write, list 等）
- **作用**: 定义智能体可用的工具接口规范

#### L3: Skills Registry - 技能注册层
- **功能**: 自动目录扫描，加载自定义技能的系统提示词片段
- **潜在问题**: 自动扫描可能带来性能与延迟开销

#### L4: Model Aliases - 模型切换层
- **功能**: 模型快捷别名映射（gln-5, opus, sonnet）
- **作用**: 简化模型调用，支持动态切换

#### L5: Protocol Specs - 通信协议层
- **功能**: 静默回复、心跳机制、回复标签等底层通信规则
- **作用**: 规范智能体与外部系统的交互协议

#### L6: Runtime Info - 运行时状态层
- **功能**: 实时状态仪表盘（主机、操作系统、模型等）
- **作用**: 提供运行时环境感知能力

**核心挑战**: 黑盒特性使得调试和溯源困难，用户必须信任底层约束。

### 2.2 用户静态控制层 (L7)

#### L7: Workspace Files - 工作区文件层
- **内容**: `IDENTITY.md`（身份/目标）、`AGENTS.md`（规则）、`MEMORY.md`（记忆）等
- **特点**: 用户完全可控，直接编辑
- **潜在问题**: 静态文件易变冗长，导致指令重复与冲突

### 2.3 用户动态注入层 (L8)

#### L8: Bootstrap Hook - 可编程注入层
- **功能**: `agent:bootstrap`、`bootstrap-extra-files`、`before_prompt_build` 等动态脚本机制
- **容量控制**: 通过 `bootstrapMaxChars` 进行容量管理
- **注入方式**:
  - `bootstrap-extra-files`: 简单的静态文件补充
  - `agent:bootstrap`: 复杂的初始化逻辑
  - `before_prompt_build`: 强实时控制需求

- **潜在问题**: 注入方式选择困难，存在滥用风险

### 2.4 实时流量接入层 (L9)

#### L9: Inbound Context - 实时流量层
- **内容**: 消息元信息、发送者信息、实时对话历史
- **特点**: 直接接触外部输入
- **潜在问题**: 缺乏防范恶意指令注入的明确机制

## 3. 关键问题回应

### 3.1 冲突与优先级机制（L7 vs L8）

**问题**: L7 静态文件与 L8 动态脚本发生冲突时，谁优先？

**结论**: ✅ **L8 优先级更高**

根据动态注入器（如 `before_prompt_build`）的特性，它在提示词构建的最后环节动态组装上下文，时间顺序晚于 L7 静态文件加载，具备覆盖低优先级静态设定的能力。

### 3.2 上下文容量与截断策略（bootstrapMaxChars）

**问题**: 当 9 层架构叠加海量前置上下文时，如何管理 Token 容量？

**结论**: ⚠️ **智能截断，保护核心层**

**推荐截断优先级**:
1. **绝对保留**: L1-L6（核心框架）
2. **优先保留**: L7 核心身份
3. **可截断**: L8 外挂文档
4. **滑动截断**: L9 早期历史

### 3.3 L8 权限边界与安全风险

**问题**: L8 是否可以修改或删除 L1-L6 的内置提示词？

**结论**: ❌ **L8 不应具备越权修改能力**

- L8 的脚本执行环境应受到严格的沙箱限制
- 其越权修改能力仅局限于重塑 L7 到 L9 的用户层上下文
- 无权篡改或删除 L1-L6 框架底层生成的内置系统提示词
- L8 和 L9 亟需引入显式的 Prompt Injection 防范机制

## 5. 优化建议

### 5.1 针对上下文膨胀 (L7 优化)

**问题**: L7 静态文件容易变得冗长，导致上下文膨胀和指令冲突。

**建议**:
- 严格执行"Refine, use checklists, avoid repeats"原则
- 定期重构 `IDENTITY.md` 和 `AGENTS.md`
- 使用精简的清单体替代长篇大论
- 避免与 L8 动态内容发生语义重复
- 每月审查 L7 文件，删除过时内容

### 5.2 针对动态注入混乱 (L8 规范化)

**问题**: L8 提供多种注入方式，容易被滥用或误用。

**建议**:
- **场景化分级注入**:
  - 简单静态文件补充 → `bootstrap-extra-files`
  - 复杂初始化逻辑 → `agent:bootstrap`
  - 强实时控制需求 → `before_prompt_build`
- **引入监控面板**: 量化 L3/L8 的性能开销，提供可视化上下文使用情况

### 5.3 针对安全防护与权限边界加固

**问题**: 安全边界模糊，存在 Prompt Injection 风险。

**建议**:

#### 固化黑盒防篡改
- 在代码层面强制隔离 L1-L6
- 确保 L1-L6 的提示词字符串对 L8 脚本完全只读
- 阻断越权修改路径

#### 抗注入过滤
在 L9 接收实时流量和 L8 挂载外部脚本之前，增加专门的安全清洗层：
- 检测并忽略任何试图重写 L1-L2 的指令
- 识别常见的 Prompt Injection 模式
- 实施输入验证和内容过滤

#### 明确截断策略透明度
- 公布 `bootstrapMaxChars` 的确切智能截断顺序
- 提供截断日志
- 显示当前上下文使用情况
