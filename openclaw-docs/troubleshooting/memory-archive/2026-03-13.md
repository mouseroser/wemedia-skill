## ✅ 固化结论：subagent announce 与 message 是两条独立链路（2026-03-13 16:30）

### 调查背景
在星鉴流水线首次完整运行后，发现：
- main 可以稳定补发监控群消息
- subagent completion/announce 能回到 main
- 但各 agent 自己声称已发送的 Telegram 消息，在真实 `sessions_spawn(mode=run)` 流程里大量缺失

### 最小复现验证
执行最小测试：
1. spawn 一个 gemini subagent（mode=run）
2. 要求它向监控群发送 start 消息
3. 写一个测试文件
4. 再向监控群发送 done 消息
5. 正常结束

**验证结果**：
- ✅ 子 agent 正常完成
- ✅ 测试文件成功写入：`agents/gemini/reports/subagent-message-test-20260313.txt`
- ✅ main 收到 completion/inter-session announce
- ❌ Telegram 发送日志中没有抓到这次 subagent 向监控群发送的对应 sendMessage 记录

### openclaw-docs notebook 结论
通过 notebooklm skill 查询 `openclaw-docs` notebook，得到结论：
1. `sessions_spawn(mode=run)` 的 subagent completion/announce 与 `message(...)` 是**两条独立链路**
2. subagent 可以**成功完成 + announce 回 parent/main**，同时它自己的 `message(...)` 仍可能不投递
3. announce 有独立的 parent-handoff / retry / queue 机制，普通 `message(...)` 不走这套回传兜底
4. **官方可靠模式**：关键通知不要依赖 subagent 自推；应由 parent/main 统一 relay/broadcast

### 最终结论
- ✅ `announce -> main` 是更可靠的链路
- ❌ `subagent message(...) -> Telegram` 不能当作可靠主链路
- fallback / timeout / auth error 会放大问题，但不是唯一原因
- 真实流水线里：**main 统一补发应作为主链路，agent 自推仅作为 best-effort 辅链路**

### 已落地修正
1. 更新 `AGENTS.md`：
   - main 发往监控群 / 晨星 DM = 可靠主链路
   - agent 自己的职能群通知 = best-effort 辅链路
2. 更新 `~/.openclaw/skills/stareval/SKILL.md`：
   - 关键进度不再依赖子 agent 自推
   - 只要文件落地 + completion/announce 回主链，main 就应立即补发监控群
