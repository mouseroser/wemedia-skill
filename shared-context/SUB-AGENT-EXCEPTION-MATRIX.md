## 核心原则

sub-agent 不是默认只许保留三件套。

更准确的说法是：
- `IDENTITY.md` / `SOUL.md` / `HEARTBEAT.md` 常常是**最小启动骨架**
- 但系统的目标是**随着使用持续进化**
- 所以 `TOOLS.md` / `MEMORY.md` / `AGENTS.md` / `BOOTSTRAP.md` / agent `memory/` 都可以按真实价值自然长出
- 服务对象信息默认收敛到 `AGENTS.md` 的“服务对象”段；只有长期确有必要时才恢复独立 `USER.md`

我们不再把这些文件看成“例外”。
我们把它们看成：
**成熟 agent 在长期使用中逐步长出来的有效器官。**

---

## 文件成长槽位

| 文件 | 什么时候值得出现 | 它解决什么问题 |
|------|------------------|----------------|
| `AGENTS.md` 的“服务对象”段 | 几乎所有 sub-agent 默认适用 | 收敛服务对象背景 / 偏好 / 稳定硬规则 |
| `USER.md` | 仅在该 agent 长期直接面向服务对象且内容明显长厚时 | 把特别厚的服务对象上下文从 `AGENTS.md` 中拆出 |
| `TOOLS.md` | 该 agent 有稳定专属工具链坑位 | 减少重复踩 CLI / proxy / path / schema 坑 |
| `MEMORY.md` | 该 agent 有长期、可复发的专属伤疤 | 把高代价教训沉淀为护栏 |
| `AGENTS.md` | 该 agent 已形成稳定工作流和 handoff 规则 | 把执行 contract 固化下来 |
| `BOOTSTRAP.md` | 需要明确首次启动 / 上手 / 交接入口 | 降低重新唤醒 / 接手成本 |
| `memory/YYYY-MM-DD.md` | 该 agent 已有独立的日级连续上下文 | 保留当天作战账本与晋升池 |

---

## 判断标准

不是问：
- “这个文件是不是例外？”

而是问：
1. 它是否承载了**真实重复价值**？
2. 主语是否清晰？
3. 没有它时，是否会重复解释、重复踩坑、重复丢上下文？
4. 它是否让 Day 40 的 agent 真正比 Day 1 更强？

只要答案是“是”，这个文件就值得存在。

## 明确反对的错误理解

### 错误理解 1：文件越少越先进
错。

先进的不是“少”，而是：
- 主语清晰
- 可持续演化
- 能承接长期使用积累

### 错误理解 2：三件套是长期目标态
错。

三件套最多只是：
- 最小启动骨架
- 新 agent 的轻量起点

不是成熟多智能体系统的终态。

### 错误理解 3：多出来的文件都叫例外
错。

只要这些文件有真实价值，它们不是例外，
而是**系统成长的证据**。
