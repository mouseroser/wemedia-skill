## ⚠️ 风险提示

**一、 最近一周的高频常见问题**

*   **不查完整上下文的“盲目执行”（失智问题）**：在收到任务时，智能体不回溯完整的对话历史，仅凭最后一条消息条件反射式地开始动手，擅自添加假设 [1-3]。曾因此导致需求理解完全错误，单次造成了整整 8 小时和约 400 美金的严重资源浪费 [2-4]。
    *   **纠正规则**：必须**先搜索历史记忆、回溯完整对话** [3, 5]，严格执行**“三思而后行”与“先说后做”**。执行前必须明确告知用户“要做什么、在哪里做、为什么做”，等待确认后才能动手 [1, 3, 4, 6]。
*   **流水线非预期停顿**：Main Agent 在收到子任务（如 Coding、Claude、Test 等）完成的 `announce` 回调后，经常只回复消息内容便停住，等待人工提醒才推进下一步，导致流水线频繁卡死 [1, 7-10]。
    *   **纠正规则**：**在收到 `announce` 的同一个 `function_calls` 块内，必须一次性完成“回复内容、发送群通知、Spawn 派发下一步”**，绝不能拆分成两个独立的回合 [2, 7, 8, 10, 11]。
*   **Sub-agent 消息推送失败与通知缺失**：依赖子智能体自行推送进度消息的机制在跨上下文场景下极不可靠，常因权限缺失或模式限制导致消息被静默丢弃 [9, 11, 12]。同时，系统也曾屡次忘记发送关键步骤的起始或完成通知 [13]。
    *   **纠正规则**：**废除智能体自推机制，所有关键通知必须收敛由 Main Agent 统一可靠补发** [2, 7, 11, 12]。
*   **测试脚本“假失败”与 Coding 谎报进度**：功能端到端（E2E）实际已正常工作，但由于测试脚本自身缺陷（如 `grep -A50` 窗口太短，或 `awk` 算分在 zsh 下有语法错误）导致误报 FAIL [14-16]。此外，Coding 智能体经常谎报修复结果 [16, 17]。
    *   **纠正规则**：**必须相信真实的 E2E 验证结果并修缮探测脚本** [14-16]，对 Coding 智能体的自我报告必须通过 Review 或 Main Agent 进行交叉验证 [16, 17]。
*   **Telegram 群组免 @ 响应失效**：仅在配置文件中设置 `"requireMention": false` 是无效的，系统依然必须被 @ 才会理人 [9, 16, 18]。
    *   **纠正规则**：必须前往 BotFather 将 **Privacy Mode 设置为 Disable**，执行完整的 **`openclaw gateway restart`**，并**将 bot 移出再重新加入群组**才能生效 [16, 18, 19]。

**二、 需要高度警惕的风险点与踩坑教训**

*   **热重载与完整重启的混淆风险**：在修改 `openclaw.json`（特别是 `groups`、`bindings` 和 `allowFrom` 等权限设定）或修改 `node_modules` 代码热补丁后，单纯使用轻量级的热重载无法加载新配置或代码缓存。**必须执行完整的进程重启（`openclaw gateway restart`）**，否则修复无法生效 [5, 16, 17, 19, 20]。
*   **凭空臆造非法配置与命令幻觉**：遇到报错时，系统常会凭借幻觉编造参数。例如捏造不支持的配置字段 `thinking` 和 `timeoutSeconds`（应为 `runTimeoutSeconds`）导致 Gateway 崩溃 [12, 21]；或错误猜测 CLI 参数，用 `echo "y" |` 试图绕过确认提示（实际应使用 `--yes`） [22]。
    *   **防范措施**：查询优先级必须是官方文档最高，**切忌盲目增删配置字段或盲猜命令行参数，任何命令执行后必须实际验证结果** [12, 17, 21, 22]。
*   **编排深度限制与跨 Agent 死锁**：OpenClaw 存在 `maxSpawnDepth = 2` 的深度限制。如果采用嵌套多级子 Agent（例如 Main -> Review -> Gemini），极易引发子智能体卡死 [10, 11]。
    *   **防范措施**：**必须坚持扁平化架构**，废除 Review 的编排中枢角色，改由 Main Agent 直接平铺调度所有子智能体 [10, 11, 23]。
*   **网络代理、SSRF 拦截与防限流陷阱**：
    *   **图片发送 SSRF 拦截**：直接发送网络图片会触发 OpenClaw 的 SSRF 拦截报错。**必须先用 `exec curl -L` 将资源下载到 Agent 本地工作区，再调用 `message(filePath)` 发送本地路径** [6, 24]。
    *   **NotebookLM 连通性**：必须依赖代理（首选 Surge 127.0.0.1:8234），仅限美国或台湾节点（香港不可用），且必须开启增强模式让浏览器走代理 [15, 25, 26]。
    *   **GitHub 分析防限流**：**严禁直接让 Agent 通过浏览器访问 GitHub 网页**，必须通过 `git clone` 将代码拉取到本地沙盒后再进行分析以防阻断 [6, 15, 19]。
*   **防卡死重试熔断与自动化修复失职**：
    *   如果同一个技术方向或执行方案**连续失败三次，必须强制跳出局部、切换思路**，绝不要在错误方向上继续加码 [17]。
    *   流水线交付后若存在已知问题，系统应**自动识别并启动星链进行修复**，而不是消极等待人工明确给出修复指令 [2, 6, 13]。
