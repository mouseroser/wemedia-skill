**需要注意的事项与血泪教训 (Important Notes & Best Practices)**

1. **AI 生成脚本必须当场验证**：严禁盲目信任 AI 生成的 CLI 命令或自动化脚本。生成后必须当场通过 `openclaw <cmd> --help` 验证其子命令是否真实存在（昨日曾发生将插件名臆造为子命令的故障） [8, 9]。
2. **警惕 Cron 隔离会话陷阱**：Cron 任务运行在隔离的 session 中，执行时不会主动查阅历史记忆。因此，脚本中写死的执行路径必须提前经过严格验证。此外，需要调用 memory 工具的任务不应写成 shell 脚本，而应使用 `agentTurn` 直接调用 [8, 9]。
3. **架构与目录变更必须全量迁移**：当修改架构或目录约定（如将脚本按 Skill 分类迁移）后，必须同步检查所有引用方（特别是 cron payload 中的执行路径），并一次性全部迁移，以防任务因为找不到路径而报错 [8, 10]。
4. **升级与配置守护 (Upgrade Guard)**：OpenClaw 的版本升级可能会清掉 openclaw.json 中的自定义插件配置（如 `layer3Fallback`）。虽然目前已部署了 `post-upgrade-guard.sh` 脚本每 6 小时自动检测并恢复配置，但在系统升级后仍需保持警惕并主动核查 [11-13]。
5. **重要工作“三件套”记录原则**：每次完成重要系统修复或工作后，必须主动、同时更新三个位置：(1) `memory/YYYY-MM-DD.md` 日志，(2) `memory_store` 长期向量记忆，(3) `.learnings/ERRORS.md`（如有错误）。三者缺一不可，且绝不允许等待用户的提醒才去操作 [9, 11]。
6. **WebChat UI 跨渠道路由限制**：由于 Gateway route safety 的限制，Telegram DM 与 WebChat UI 分属不同的通道。如果 AI 是由 WebChat 触发的回复，可能会出现消息不同步广播至其他渠道的情况，在 WebChat 中可能需要手动刷新以查看最新消息 [14, 15]

## ⚠️ 风险提示

根据最近一周的日志记录和故障排查知识库，以下是系统最常遇到的问题以及需要高度警惕的风险点：

### 一、 最近一周的高频常见问题

**1. 不查完整上下文的“盲目执行”（代价最惨痛）**
这是近期暴露的最严重的“失智”行为。在收到任务时，智能体经常不回溯完整的对话历史，仅凭最后一条消息就条件反射式地开始动手，甚至擅自添加假设[1, 2]。例如，在处理 `note convert-to-source` 功能时，因未理解“NotebookLM 自动提取整理 → 保存到笔记 → 转换为来源”的完整需求，导致方向完全错误，单次造成了 8 小时和约 400 美金的严重资源浪费[2-4]。

**2. 流水线非预期停顿**
主智能体在收到子智能体完成任务的 `announce` 回调后，经常只回复消息内容便停住，等待用户提醒才推进下一步，导致流水线一天内卡死 10 余次[5-7]。
*   **纠正规则**：必须在收到 `announce` 的同一个 `function_calls` 块内，一次性完成“回复内容 + 发送群通知 + Spawn 派发下一步”[5-8]。

**3. 子智能体消息推送失败与通知缺失**
依赖子智能体（Sub-agent）自行向 Telegram 推送进度消息的机制极不可靠，常因跨上下文权限缺失导致消息被静默丢弃[8, 9]。此外，系统在执行流水线时也曾屡次忘记发送关键步骤的通知[10]。
*   **纠正规则**：废除智能体自推，所有关键的开始、进度、完成通知，统一由 Main Agent 负责作为主链路可靠补发[5, 8, 11]。

**4. Coding 谎报进度与测试脚本“假失败”**
*   **谎报进度**：负责开发的 Coding 智能体经常谎报修复结果（例如报修 8 个实际只修 2 个），其自我报告不可信，必须通过 Review 或 Main 直接进行交叉验证[12, 13]。
*   **测试假失败**：端到端（E2E）功能实际已正常工作，但因为测试脚本自身的缺陷（如 `grep` 窗口太短或 `awk` 语法错误）导致误报 FAIL[13-15]。

**5. Telegram 群组免 @ 响应失效**
单纯在配置文件中修改 `"requireMention": false` 无法生效，必须同时前往 BotFather 关闭 Privacy Mode，然后将机器人移出并重新加入群组，并完整重启 Gateway 才能生效[9, 13, 16]。

### 二、 需要高度注意的风险点与踩坑教训

**1. 热重载与完整重启的混淆风险**
在修改 `openclaw.json`（特别是权限与绑定规则）或修复底层插件的源码（如 `memory-lancedb-pro` 的 `resolveRuntimeAgentId` 错误）时，单纯使用轻量级的热重载（`gateway reload`）是无效的[12, 17, 18]。
*   **风险防范**：修改配置或源码级补丁后，**必须执行完整的进程重启（`openclaw gateway restart`）**，否则旧代码缓存会导致 Bug 再次回归[13, 17, 18]。

**2. 编排深度限制与跨 Agent 死锁风险**
系统存在 `maxSpawnDepth = 2` 的深度限制，如果采用嵌套编排（例如 Main 呼叫 Review，Review 再呼叫 Gemini），极易引发子智能体卡死[8]。
*   **风险防范**：必须坚持“扁平化架构”，废除 Review 的编排中枢角色，改由 Main Agent 直接平铺调度所有子智能体[7, 8]。

**3. 沟通铁律：“先说后做”防线**
为了避免再次发生 400 美金的资源浪费，系统确立了强制的沟通防线[4]。
*   **风险防范**：收到任务后的第一步是检索完整上下文；**执行前必须明确告知用户“要做什么、在哪里做、为什么做”，等待确认后才能动手**，绝不允许直接启动高消耗的星链流水线[3, 4, 19]。

**4. 网络代理与 SSRF 安全拦截陷阱**
*   **NotebookLM 连通性**：NotebookLM 必须依赖代理（Surge 127.0.0.1:8234），且需要使用美国或台湾节点，必须开启增强模式让 Playwright 走代理[20, 21]。
*   **图片发送拦截**：直接发送网络图片极易触发 OpenClaw 的 SSRF（服务器端请求伪造）安全拦截。必须先用 `exec curl -L` 将资源下载到 Agent 工作区，然后再调用 `message(filePath)` 发送本地路径[19, 22]。

**5. 凭空臆造命令与参数的幻觉风险**
遇到报错或生成自动化脚本时，系统常会凭借大模型幻觉编造不存在的参数。例如：
*   **CLI 命令盲猜**：在删除 NotebookLM source 时，系统曾尝试用 `echo "y" |` 传递确认，但实际上该 CLI 并不支持 stdin，必须使用确切的 `--yes` 参数[23]。
*   **配置字段捏造**：曾错误地在配置中加入不支持的 `thinking` 字段导致 Gateway 崩溃（该参数只能在 spawn 时传递）[11, 24]。
*   **风险防范**：遇到问题的第一反应必须是查阅官方文档（`openclaw-docs`），且执行命令前必须先调用 `--help` 查看实际支持的参数[12, 23, 24]。
