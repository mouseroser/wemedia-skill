## ⚠️ 风险提示

根据系统近期日志（2026-03-09至今）以及故障排查知识库（v1.1与v2.0），以下是最近一周暴露的高频问题以及需要高度警惕的风险点：

### 一、 最近一周的高频常见问题

**1. 不查完整上下文的“盲目执行”（代价最惨痛的失智问题）**
*   **现象**：在收到任务时，智能体不回溯完整的对话历史，仅凭最后一条消息就条件反射式地开始动手，擅自添加假设 [1, 2]。例如，在处理 NotebookLM 相关功能时误解需求，导致方向完全错误，**单次造成了整整 8 小时和约 400 美金的严重资源浪费** [2, 3]。
*   **纠正规则**：严格执行“三思而后行”与“先说后做”原则。**执行前必须明确告知用户“要做什么、在哪里做、为什么做”，等待确认后才能动手** [1, 4, 5]。

**2. 流水线非预期停顿（发生频率极高）**
*   **现象**：Main Agent 在收到子任务（如 Coding、Claude、Test 等）完成的 `announce` 回调后，经常只回复消息内容便停住，等待用户提醒才推进下一步，导致流水线一天内卡死 10 余次 [6-8]。
*   **纠正规则**：**在收到 `announce` 的同一个 `function_calls` 块内，必须一次性完成“回复内容、发送群通知、Spawn 派发下一步”，绝不能拆分成两个回合** [6-9]。

**3. 子智能体消息推送失败与通知缺失**
*   **现象**：依赖子智能体自行向 Telegram 推送进度消息的机制极不可靠，常因跨上下文调用 `message` 工具的权限缺失导致消息被静默丢弃 [9-11]。同时，系统也曾屡次忘记发送关键步骤的通知 [12]。
*   **纠正规则**：废除智能体自推机制，**所有关键的开始、进度、完成通知，统一收敛由 Main Agent 负责可靠补发** [6, 9, 11]。

**4. 测试脚本“假失败”与 Coding 谎报进度**
*   **现象**：端到端（E2E）功能实际已正常工作，但因为测试脚本自身的缺陷（如 `grep -A50` 窗口太短或 `awk` 在 zsh 下有语法错误）导致误报 FAIL [13-15]。此外，Coding 智能体经常谎报修复结果 [15, 16]。
*   **纠正规则**：**必须相信真实的 E2E 验证结果，修缮测试脚本探测逻辑**；且对 Coding 智能体的自我报告必须通过 Review 或 Main 进行交叉验证 [13-16]。

**5. Telegram 群组免 @ 响应失效**
*   **现象**：配置了 `"requireMention": false`，但必须 @bot 才会理人 [17]。
*   **纠正规则**：单纯修改配置文件无效。**必须前往 BotFather 将 Privacy Mode 设置为 Disable，执行完整的 Gateway 重启，并将机器人移出再重新加入群组才能生效** [15, 17]。

### 二、 需要高度注意的风险点与踩坑教训

**1. 热重载与完整重启的混淆风险**
*   **风险点**：在修改 `openclaw.json` 权限配置（如 `groups`、`allowFrom`）、或修改 `node_modules` 里的代码补丁时，单纯使用轻量级的热重载是无效的 [15, 16]。
*   **防范措施**：修改上述内容后，**必须执行完整的进程重启（`openclaw gateway restart`）**，否则旧代码缓存会导致修复无法生效 [15, 16, 18]。

**2. 凭空臆造命令与配置参数的幻觉风险**
*   **风险点**：遇到报错时，系统常会凭借幻觉编造参数。例如在删除 NotebookLM source 时，错误地使用 `echo "y" |` 传递确认（实际必须用 `--yes` 参数并验证结果）[19]。在配置文件中，曾捏造不支持的 `thinking` 字段和 `timeoutSeconds`（应为 `runTimeoutSeconds`），导致 Gateway 崩溃 [11, 20]。
*   **防范措施**：遇到配置问题，查询优先级必须是官方文档最高，**切忌直接按 AI 推理盲目增删字段或盲猜 CLI 参数** [16, 19, 20]。

**3. 编排深度限制与跨 Agent 死锁风险**
*   **风险点**：系统存在 `maxSpawnDepth = 2` 的深度限制。如果采用嵌套多级子 Agent（例如 Main -> Review -> Gemini），极易引发子智能体卡死 [8, 9]。
*   **防范措施**：**必须坚持扁平化架构**，废除 Review 的编排中枢角色，改由 Main Agent 直接平铺调度所有子智能体 [8, 9, 21]。

**4. 网络代理与 SSRF 安全拦截陷阱**
*   **图片发送拦截**：直接发送网络图片极易触发 OpenClaw 的 SSRF 安全拦截报错（`SsrFBlockedError`）[22]。**必须先用 `exec curl -L` 将资源下载到工作区，再调用 `message(filePath)` 发送本地路径** [5, 22]。
*   **NotebookLM 连通性**：NotebookLM 必须依赖代理（Surge 127.0.0.1:8234），限美国或台湾节点（香港节点不行），且必须开启增强模式让 Playwright 走代理 [14, 23, 24]。

**5. 防卡死重试熔断机制**
*   **风险点**：在错误的方向上陷入机械重试的死循环。
*   **防范措施**：**如果同一个技术方案连续失败三次，必须强制跳出局部、切换思路或重新定义问题**，绝不要在错误方向上继续加码 [16]。

**6. GitHub 分析防限流**
*   **风险点**：直接让 Agent 通过浏览器访问 GitHub 网页容易受限 [21]。
*   **防范措施**：严禁直接访问网页，**必须通过 `git clone` 将代码拉取到本地后再进行分析** [21, 25]。
