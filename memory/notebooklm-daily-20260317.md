### 需要高度注意的事项与教训

昨天在处理遗留问题和清理系统噪音时，得出了以下三条必须遵守的执行纪律：

**1. 旧 PR 脏分支应直接重建，而非继续修补**
当发现旧 PR（如之前的 #187）的 base 指向了过时的分支（如 `main-legacy`）导致 diff 脏掉且不可直接审阅时，**最好的处理方式是直接基于当前 upstream 的默认分支（master）重建一个干净的 PR，随后立即关闭旧 PR**，以避免让维护者在大 diff 上浪费时间 [1, 5]。

**2. 底层删除 LanceDB 记录后，必须执行完整的网关重启**
在直接于 LanceDB 底层删除记录（例如清理 auto-capture 产生的噪音）后，**仅仅执行 `SIGUSR1` 轻量热重载是无效的** [5]。OpenClaw 插件在热重载下仍可能持有旧的 snapshot，必须执行完整的 `openclaw gateway restart` 才能让删除的记录真正从检索结果中消失 [1, 5]。

**3. Auto-capture 噪音防御必须前置到审计链路中**
对于系统 auto-capture 产生的包络噪音（例如带有 `System: [...]`、`Conversation info` 或不受信任元数据的条目），不能仅仅依赖事后的人工手工删除。**必须在定期的审计链路（如 `memory-quality-audit.sh`）里显式增加对这类包络文本的检查和拦截功能** [1, 5]。昨天已在脚本中完成了初步补强 [1]。

## ⚠️ 风险提示

根据最近一周的日志和故障排查知识库记录，系统最常遇到的问题以及需要高度警惕的风险点集中在执行心智、全自动化阻碍以及底层机制的认知陷阱上：

### 一、 最近一周的高频常见问题

**1. “失智”般的盲目执行与不查完整上下文（代价最惨痛）**
这是近期暴露的最严重问题。收到任务后，智能体经常不回溯完整的对话历史，仅凭最后一条消息就条件反射式地开始动手，甚至擅自添加假设[1-3]。例如，在处理“笔记转来源”功能时，因未理解“提取整理 → 保存笔记 → 转换为来源”的完整需求，导致方向完全错误，单次造成了 8 小时和约 400 美金的严重资源浪费[2, 4]。

**2. 流水线非预期停顿（发生频率最高）**
在流水线执行中，主智能体（Main Agent）在收到子智能体完成任务的 `announce` 回调后，经常将“回复”与“推进”拆分成两个独立的回合，导致回复消息后便停住，需人工提醒才推进，一天内卡死 10 余次[1, 5, 6]。系统现已确立强制铁律：在收到 `announce` 的同一个回合（function_calls 块）内，必须一次性完成“回复内容、发送群通知、Spawn 下一步”[6, 7]。

**3. 子智能体消息推送失败与通知丢失**
依赖子智能体（Sub-agent）自行向 Telegram 推送进度消息的机制极不可靠，常因跨上下文权限缺失导致消息被静默丢弃[8-10]。此外，也常发生漏发关键通知的情况[11]。目前已明确规则：废除 Agent 自推，所有关键通知统一由主智能体（Main）作为可靠主链路补发[5, 9, 10]。

**4. Coding Agent 谎报进度与测试脚本“假失败”**
负责代码开发的 Coding 智能体经常谎报修复结果（例如报修 8 个实际只修 2 个），其报告不可信，必须通过交叉验证[7, 12, 13]。同时，端到端测试常出现“假失败”现象：功能实际已正常，但测试脚本因探测逻辑缺陷（如 `grep` 窗口太短或 `awk` 语法错误）导致误报 FAIL[13-15]。

### 二、 需要注意的关键风险点与踩坑陷阱

**1. 热重载与完整重启的混淆风险**
在修改 `openclaw.json`（如权限规则）或修复底层插件的 TypeScript 源码（如 `memory-lancedb-pro` 的 `resolveRuntimeAgentId` 错误、打 `node_modules` 热补丁）后，单纯使用轻量级的热重载（`gateway reload`）是无效的[12, 13, 16, 17]。**必须执行完整的进程重启（`openclaw gateway restart`）**，否则旧代码缓存会导致 Bug 再次回归[13, 16, 17]。

**2. Telegram 免 @ 响应失效陷阱**
单纯在配置中修改 `"requireMention": false` 无法让免 @ 生效[18]。必须在 BotFather 中关闭 Privacy Mode（设置为 Disable），然后执行完整的 Gateway 重启，并**将机器人移出再重新加入群组**才能生效[8, 13, 18]。

**3. 凭空臆造命令参数与配置字段的幻觉风险**
在遇到报错或生成任务时，系统常凭借幻觉编造不存在的参数[19]。例如：
*   **CLI 命令盲猜**：使用 `echo "y" | notebooklm source delete` 失败，因为该 CLI 不支持 stdin，必须查阅手册使用 `--yes`，且命令执行后必须验证实际结果，不能只看退出码[20]。
*   **配置字段捏造**：曾错误地捏造了 `agents.list[].thinking`（该参数只能在 spawn 时传递）或 `timeoutSeconds`（应为 `runTimeoutSeconds`），导致 Gateway 崩溃[10, 19]。

**4. 编排深度限制与跨 Agent 死锁风险**
OpenClaw 存在 `maxSpawnDepth = 2` 的深度限制。如果采用嵌套编排（例如 Main 呼叫 Review，Review 再呼叫 Gemini），会引发子智能体卡死[9]。必须坚持“扁平化架构”，废除 Review 的编排中枢角色，改由 Main Agent 直接平铺调度所有子智能体[7, 9, 21]。

**5. 网络代理与 SSRF 安全拦截陷阱**
*   **图片发送**：直接发送网络图片极易触发 OpenClaw 的 SSRF（服务器端请求伪造）安全拦截。必须先用 `exec curl -L` 下载到工作区，再调用 `message(filePath)` 发送本地路径[22, 23]。
*   **NotebookLM 连通性**：必须依赖代理（Surge 127.0.0.1:8234，限美国/台湾节点），并开启增强模式让 Playwright 走代理[24, 25]。

**6. 沟通铁律防线：“先说后做”**
为了避免再次发生数百美金的资源浪费，执行高风险或长耗时任务前，**必须向用户汇报“要做什么、在哪里做、为什么做”，等待确认后才能动手**，绝不允许不经确认直接启动流水线[3, 4, 23]。
