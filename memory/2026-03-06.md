## Runtime/Workspace 分离

### 问题发现
部分子智能体的 workspace 下存在 `agent/`、`sessions/` 目录：
- brainstorming, gemini, wemedia, monitor-bot

### 解决方案
将所有 runtime 目录从 workspace 移动到根级 agents：

**移动操作**:
- `~/.openclaw/workspace/agents/{id}/agent/` → `~/.openclaw/agents/{id}/agent/`
- `~/.openclaw/workspace/agents/{id}/sessions/` → `~/.openclaw/agents/{id}/sessions/`

**原因**:
- `~/.openclaw/agents/*` = OpenClaw 内核 runtime（Engine Layer）
- `~/.openclaw/workspace/*` = 主智能体/子智能体工作区（Workspace Layer）
- 职责分离，安全性更高，备份/同步更清晰

### TOOLS.md 更新
所有智能体的 TOOLS.md 添加说明：
> 若本目录下存在 agent/、sessions/，视为 OpenClaw 内核 runtime 子目录，不参与流水线，请勿修改。

## notebooklm-py 认证根因（2026-03-06 16:32）
- 浏览器里看似已登录 NotebookLM，但 `notebooklm-py` 程序请求仍会被重定向到 Google 账号选择页
- 根因不是“没登录”，而是 `notebooklm-py` 在取 token 时把浏览器保存的 cookies 压平成普通 header/dict，丢失了 cookie 的域名作用域
- 结果：浏览器可见登录态 ≠ CLI 可编程认证态；Google 跨域跳转到 `accounts.google.com` 时，程序拿不到有效 token
- 已做本地补丁：在 `notebooklm-py` 的 `auth.py` 中，改为使用保留域名作用域的 `httpx.Cookies` jar 获取 token，而不是用压平后的 cookie header
- 验证结果：`notebooklm auth check --test` 通过，`nlm-gateway.sh query --agent main --notebook openclaw-docs ...` 成功

## 经验结论
- NotebookLM 登录问题不能只看“浏览器是否进了首页”，必须看 `auth check --test` 是否真的能拿到 token
- 当 Google / NotebookLM 依赖跨域 cookie 时，不能把 Playwright 存下来的 cookies 简单压平成一串 header；必须保留 domain/path 作用域
- `openclaw-docs` 若被删除重建，除了更新 skill registry，还要重新验证 query/research 主链路是否都仍然正常
