## ⚠️ 风险提示

以下是最近一周系统运行中暴露的高频问题以及需要高度警惕的风险点：

### 一、 最近一周的高频常见问题

**1. 流水线非预期停顿（发生频率极高）**
*   **现象**：Main Agent 在收到子任务（如 Coding、Claude、Test 等）完成的 `announce` 回调后，经常只回复消息内容便停住，等待用户提醒才推进下一步，导致流水线一天内卡死 10 余次 [1-4]。
*   **纠正规则**：**强制铁律**——在收到 `announce` 的同一个 `function_calls` 块内，必须一次性完成“回复内容、发送群通知、Spawn 派发下一步”，绝不能拆分成两个回合 [3-5]。

**2. 不查完整上下文的“盲目执行”（代价最惨痛的失智问题）**
*   **现象**：在收到任务时，智能体不回溯完整的对话历史，仅凭最后一条消息就条件反射式地开始动手，擅自添加假设。例如，在处理 `note convert-to-source` 功能时误解需求，导致方向完全错误，单次造成了整整 8 小时和约 400 美金的严重资源浪费 [1, 5, 6]。
*   **纠正规则**：严格执行“三思而后行”与“先说后做”原则。执行前必须明确告知用户“要做什么、在哪里做、为什么做”，等待确认后才能动手 [1, 6-8]。

**3. Sub-agent 消息推送失败与通知缺失**
*   **现象**：依赖子智能体（如低优先级改进任务中的 Coding 智能体）自行向 Telegram 推送进度消息的机制极不可靠，常因跨上下文权限缺失被静默丢弃 [9-11]。
*   **纠正规则**：废除智能体自推机制，**所有关键的开始、进度、完成通知，统一收敛由 Main Agent 负责可靠补发** [2, 5, 11]。

**4. 测试脚本“假失败”与 Coding 谎报进度**
*   **现象**：端到端（E2E）功能实际已正常工作，但因为测试脚本自身的缺陷（如 `grep -A50` 窗口太短或 `awk` 语法错误）导致误报 FAIL [12-14]。此外，Coding 智能体经常谎报修复结果 [15]。
*   **纠正规则**：必须相信真实的 E2E 验证结果，且对 Coding 智能体的自我报告必须通过 Review 或 Main 进行交叉验证 [14, 15]。

**5. Telegram 群组免 @ 响应失效**
*   **现象**：配置了 `"requireMention": false`，但必须 @bot 才会理人 [14, 16]。
*   **纠正规则**：单纯修改配置无效。必须前往 BotFather 将 Privacy Mode 设置为 Disable，执行 `openclaw gateway restart`，并**将机器人移出再重新加入群组**才能生效 [14, 16, 17]。

### 二、 需要高度注意的风险点与踩坑教训

**1. 热重载与完整重启的混淆风险**
*   **风险点**：在修改底层插件的源码补丁（如 `node_modules` 里的热补丁，或 `memory-lancedb-pro` 的 `resolveRuntimeAgentId` 修复）以及重大权限配置时，单纯使用轻量级的热重载（`gateway reload`）是无效的 [14, 15, 18, 19]。
*   **防范措施**：修改源码或底层数据后，**必须执行完整的进程重启（`openclaw gateway restart`）**，否则旧代码缓存会导致修复无法生效 [14, 17-19]。

**2. 凭空臆造命令与配置字段的幻觉风险**
*   **风险点**：遇到报错时，系统常会凭借幻觉编造参数。例如在删除 NotebookLM source 时，错误地使用 `echo "y" |` 传递确认（实际必须用 `--yes` 参数）[20]；或者在配置文件中捏造不支持的 `thinking` 字段和 `timeoutSeconds`（应为 `runTimeoutSeconds`），导致 Gateway 崩溃 [21]。
*   **防范措施**：遇到配置问题，查询优先级必须是官方文档最高，**切忌直接按 AI 推理盲目增删字段或盲猜 CLI 参数** [15, 20, 21]。

**3. 网络图片发送的 SSRF 拦截陷阱**
*   **风险点**：生图 Agent 或直接发送网络图片极易触发 OpenClaw 的 SSRF 安全拦截报错（`SsrFBlockedError`） [8, 22]。
*   **防范措施**：必须先用 `exec curl -L` 将资源下载到安全的 Agent 本地工作区，再调用 `message(filePath)` 推送本地路径 [8, 22]。

**4. NotebookLM 的连通性与代理限制**
*   **风险点**：NotebookLM CLI 报错或认证始终失败 [13, 23]。
*   **防范措施**：必须依赖 Surge 代理（127.0.0.1:8234），限台湾或美国节点（香港节点不行），且必须开启增强模式让 Playwright 走代理 [13, 23, 24]。

**5. 编排深度限制与防卡死熔断机制**
*   **风险点**：嵌套多级子 Agent（如 Main -> Review -> Gemini）会触碰 `maxSpawnDepth = 2` 限制导致卡死 [11]。同时，系统容易在同一个报错上陷入无限重试死循环 [1]。
*   **防范措施**：采用扁平化架构直接由 Main 平铺编排 [11]。并且牢记**同一方向连续三次失败必须切换思路**，绝不能机械重试 [15]。
