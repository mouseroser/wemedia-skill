# DECISION-FRAMEWORK.md - 收益评估决策框架

## 目的

明确哪些改动可以自行执行，哪些需要先讨论。避免浪费时间在收益不明确的改动上。

---

## 🟢 可以自行执行（无需询问）

**条件**：同时满足以下所有条件
1. **明确的痛点** - 当前有实际问题（bug、错误、不可用）
2. **立即可见的收益** - 修复后立即改善（不是"可能会更好"）
3. **低风险** - 改动范围小，容易回滚
4. **低成本** - 改动时间 < 30 分钟

**例子**：
- ✅ 修复脚本 bug（--task 改为 -m）
- ✅ 清理明显无用的文件
- ✅ 修正配置错误
- ✅ 更新过时的文档
- ✅ 记录教训到 MEMORY.md

**执行后**：
- 立即记录到 memory/YYYY-MM-DD.md
- 重要的存储到 memory-lancedb-pro

---

## 🟡 需要先评估并讨论

**条件**：符合以下任一条件
1. **收益不确定** - "可能会更好"、"看起来更整洁"
2. **使用场景不明确** - 不知道多久会用到
3. **中等成本** - 改动时间 30-60 分钟
4. **影响多个组件** - 需要修改多个文件或 agent

**流程**：
1. 分析收益（具体是什么？使用场景？频率？）
2. 评估成本（改动范围？时间？风险？）
3. 提出方案并说明收益/成本
4. 等待晨星的决定

**例子**：
- 🟡 架构优化（intel/ 目录集中案例）
- 🟡 流程改进
- 🟡 新功能添加
- 🟡 工具升级

---

## 🔴 必须先讨论（绝不自行执行）

**条件**：符合以下任一条件
1. **高成本** - 改动时间 > 60 分钟
2. **高风险** - 难以回滚，影响核心功能
3. **重大架构变更** - 改变系统基本结构
4. **外部影响** - 影响其他人或系统

**例子**：
- 🔴 流水线架构重构
- 🔴 记忆系统迁移
- 🔴 Agent 职责重新分配
- 🔴 外部 API 集成

---

## 🎯 快速判断

**自行执行**：修复明确的问题，改动 < 30 分钟  
**先讨论**：优化、改进、不确定收益、改动 > 30 分钟

---

## 原则

1. **保持简单** - 除非有明确的使用场景，否则不改动
2. **尊重时间** - 不浪费晨星的时间在收益不明确的事情上
3. **先思考再行动** - 看到问题 → 想到方案 → **评估收益** → 讨论 → 再决定
4. **容易回滚** - 任何改动都应该容易回滚

---

## 历史案例

### 案例 1：流水线产物集中到 intel/ 目录（2026-03-12）
- **提议**：将各 agent 的产物集中到 intel/<pipeline>/run-*/ 目录
- **收益**：便于查看、归档、清理
- **问题**：收益不明确（没有明确的使用场景），改动成本高
- **决定**：回滚
- **教训**：架构改动前必须先评估收益

### 案例 2：修复 launcher 脚本 bug（2026-03-12）
- **问题**：--task 参数错误，脚本无法运行
- **收益**：立即可见（脚本可以运行）
- **成本**：低（5 分钟）
- **决定**：自行执行
- **结果**：✅ 成功修复并推送
