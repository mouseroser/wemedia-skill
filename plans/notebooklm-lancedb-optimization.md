# NotebookLM + memory-lancedb-pro 联动优化方案

**制定日期**: 2026-03-10  
**基于**: 三层记忆架构（已部署）  
**目标**: 让两个系统互补增强，而不是重复

---

## 当前架构回顾

### Layer 1: 原始对话层（markdown）
- `memory/YYYY-MM-DD.md` — 每日记忆
- `MEMORY.md` — 长期记忆
- **作用**: 底稿，最终依据

### Layer 2: 结构化记忆层（LanceDB）
- memory-lancedb-pro
- Vector + BM25 混合检索
- 本地 rerank sidecar
- **作用**: 快速语义检索

### Layer 3: 摘要导航层（NotebookLM）
- Memory Archive notebook
- 每天凌晨 04:00 自动同步
- **作用**: 摘要、时间线、冲突检测

---

## 联动优化方案

### 优化 1: 智能检索路由

**问题**: 现在 `memory_recall` 只查 LanceDB，NotebookLM 只能手动查询

**方案**: 建立检索路由规则

```
用户问题
  ↓
判断问题类型
  ↓
├─ 精确查询（关键词、日期）→ LanceDB（快速）
├─ 模糊查询（"晨星的偏好"）→ NotebookLM（理解力强）
├─ 冲突检测（"X 和 Y 矛盾吗"）→ NotebookLM（对比能力强）
└─ 时间线查询（"最近发生了什么"）→ NotebookLM（摘要能力强）
```

**实现方式**:
- 在 `memory_recall` 前，先判断问题类型
- 如果是模糊查询/冲突检测/时间线 → 调用 NotebookLM
- 如果是精确查询 → 使用 LanceDB

### 优化 2: 双向验证机制

**问题**: LanceDB 的提炼可能有误，但没有验证机制

**方案**: 关键决策前，用 NotebookLM 验证

```
收到关键问题（偏好、限制、决策）
  ↓
1. LanceDB 快速召回
  ↓
2. NotebookLM 验证原话
  ↓
3. 如果一致 → 使用
4. 如果冲突 → 以 NotebookLM（原话）为准
```

**触发条件**:
- 用户纠正我的理解
- 涉及重要决策
- 发现记忆冲突

### 优化 3: 自动质量审计

**问题**: 现在只有定时同步，没有主动审计

**方案**: 每天凌晨 04:00 执行三项审计

```bash
# 1. 情绪污染检测
notebooklm ask "找出最近24小时内所有情绪表达，判断是否误存入长期记忆"

# 2. 冲突检测
notebooklm ask "检查最近7天的记忆是否有冲突（同一主题的不同说法）"

# 3. 溯源完整性检查
notebooklm ask "列出所有缺少日期标记的记忆"
```

**输出**: 生成审计报告，推送到监控群

### 优化 4: 记忆分级存储

**问题**: 所有记忆都平等对待，没有优先级

**方案**: 根据重要性分级

```
高优先级（立即同步到 NotebookLM）:
- 偏好、限制、承诺、纠正
- 重要决策
- 教训总结

中优先级（每天同步）:
- 稳定事实
- 项目状态
- 实体关系

低优先级（每周同步）:
- 一般对话
- 临时信息（但这些不应该存）
```

**实现方式**:
- 在 `memory_store` 时，根据 `category` 和 `importance` 判断
- 高优先级 → 立即触发 NotebookLM 同步
- 中优先级 → 等待定时同步
- 低优先级 → 只存 LanceDB，不同步 NotebookLM

### 优化 5: 原话溯源增强

**问题**: LanceDB 记忆无法直接追溯到原话

**方案**: 在 NotebookLM 中建立索引

```
每次同步时:
1. 上传 markdown 文件到 NotebookLM
2. 生成"记忆索引"（记忆ID → 原话位置）
3. 存储到 NotebookLM 的一个专门 note

查询时:
1. LanceDB 返回记忆ID
2. 查询 NotebookLM 的"记忆索引"
3. 定位到原话的具体位置
```

## 预期效果

**检索效率**:
- 精确查询：LanceDB（<1秒）
- 模糊查询：NotebookLM（5-10秒，但更准确）

**记忆质量**:
- 自动检测情绪污染
- 自动发现冲突
- 自动验证溯源完整性

**可靠性**:
- 关键决策有双向验证
- 原话可追溯
- 冲突有明确解决规则

---

**核心理念**: NotebookLM 不是 LanceDB 的替代品，而是增强层。LanceDB 负责快速检索，NotebookLM 负责深度理解和验证。
